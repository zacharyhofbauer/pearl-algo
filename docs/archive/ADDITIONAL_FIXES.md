# Additional Fixes - Signal Count, Stop Notification, and Menu

## Issues Fixed

### 1. Signal Count Resets to 0 After Restart
**Problem**: When the agent restarts, the signal count resets to 0 even though signals were previously generated and persisted.

**Root Cause**: The `signal_count` was always initialized to 0 on startup and never restored from saved state.

**Fix**: 
- Now loads saved state on startup
- Restores `signal_count` from saved state
- Also counts actual signals from the signals file for accuracy (uses the higher of the two)
- Logs the restored count for visibility

**File Modified**: `src/pearlalgo/nq_agent/service.py`

**What You'll See**:
```
Restored signal_count: 2 (from state: 2, from file: 2)
```

### 2. No Notification When Agent Stops
**Problem**: Agent stops but no shutdown notification is sent to Telegram.

**Root Cause**: 
- Shutdown notification might fail silently
- Service might be killed before notification can be sent
- Timeout was too short (5 seconds)

**Fix**:
- Increased timeout from 5s to 10s
- Added retry logic if first attempt times out
- Enhanced error logging to show exactly what failed
- Moved notification sending BEFORE setting `running=False` to ensure Telegram is still available

**File Modified**: `src/pearlalgo/nq_agent/service.py`

**What You'll See**:
- `Sending shutdown notification: <reason>`
- `✅ Shutdown notification sent to Telegram` (success)
- OR `❌ Timeout sending shutdown notification` (with retry attempt)

### 3. Menu Not Showing in Telegram
**Problem**: `/start` command doesn't show the menu with buttons.

**Possible Causes**:
- Command handler not running
- Error sending message with buttons
- Message sent but buttons not rendering

**Fixes Applied**:
- Enhanced error handling in `_send_message_or_edit()`
- Added detailed logging for message sending
- Added fallback if `update.message` is missing
- Logs number of button rows being sent
- Better error messages if sending fails

**File Modified**: `src/pearlalgo/nq_agent/telegram_command_handler.py`

**What You'll See in Logs**:
```
Received /start command from chat <chat_id>
Sending /start menu with 7 button rows to chat <chat_id>
Sent message with 7 button rows to chat <chat_id>
```

## Troubleshooting

### If Signal Count Still Shows 0 After Restart

1. **Check if state file exists**:
   ```bash
   ls -la data/nq_agent_state/state.json
   ```

2. **Check if signals file exists**:
   ```bash
   ls -la data/nq_agent_state/signals.jsonl
   wc -l data/nq_agent_state/signals.jsonl  # Count signals
   ```

3. **Check logs for restoration message**:
   ```bash
   grep "Restored signal_count" logs/nq_agent.log
   ```

### If Stop Notification Still Not Working

1. **Check if Telegram is enabled**:
   ```bash
   grep -i "telegram.*enabled" logs/nq_agent.log | tail -5
   ```

2. **Check shutdown logs**:
   ```bash
   grep -i "shutdown" logs/nq_agent.log | tail -10
   ```

3. **Verify Telegram credentials**:
   - Check `.env` file has `TELEGRAM_BOT_TOKEN` and `TELEGRAM_CHAT_ID`
   - Restart agent to see initialization logs

### If Menu Still Not Showing

1. **Check if command handler is running**:
   ```bash
   ./scripts/telegram/check_command_handler.sh
   ```

2. **Start command handler if not running**:
   ```bash
   ./scripts/telegram/start_command_handler.sh --background
   ```

3. **Check command handler logs**:
   ```bash
   tail -f logs/telegram_handler.log
   ```

4. **Try `/start` command again** and check logs for:
   - `Received /start command`
   - `Sending /start menu with X button rows`
   - `Sent message with X button rows`
   - Any error messages

5. **Verify you're using the correct bot**:
   - Make sure you're messaging the bot that has the command handler running
   - Check `TELEGRAM_CHAT_ID` matches your chat ID

## Architecture Notes

### Signal Count Persistence
- **State File**: Stores `signal_count` in `data/nq_agent_state/state.json`
- **Signals File**: Stores actual signals in `data/nq_agent_state/signals.jsonl` (one per line)
- **On Startup**: Restores count from state file AND counts from signals file, uses the higher value
- **Why Both?**: State file might be more recent, but signals file is the source of truth

### Shutdown Notification Flow
1. `stop()` method called (via signal handler or exception)
2. Save final state to disk
3. **Send shutdown notification** (with retry if needed)
4. Set `running=False`
5. Service exits

### Menu Button Architecture
- **Command Handler**: Separate service that listens for Telegram commands
- **Main Menu**: Generated by `_get_main_menu_buttons()` with 7 rows of buttons
- **Button Types**: 
  - Service control (Start/Stop/Restart Agent)
  - Gateway control
  - Monitoring (Status, Signals)
  - Analysis (Performance, Backtest)
  - Configuration (Config, Health)
  - Testing (Test Signal)
  - Help
